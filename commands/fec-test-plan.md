description: 为前端功能、PR 或里程碑制定风险驱动的测试策略，聚焦测什么、在哪一层测、优先级和证据产物；不做架构实施规划。
---

为当前功能或变更制定前端测试策略。这个命令回答“测什么、用哪一层测、为什么这样测、失败时看什么证据”，不负责拆组件、设计数据流或安排实现阶段。

先读取现有测试工具、命令、CI 和目录约定，再按风险选择最小可维护覆盖。若需要独立规划，可委托 **`fec-test-planner`** 子代理；若用户已经明确要求写具体测试，则直接分流到 `fec-component-testing`、`fec-e2e-testing` 或 `fec-storybook-component-doc`。

## 执行步骤

1. 确定范围：
   - 用户指定功能、PR、文件或业务流程时，以该范围为准。
   - 用户未指定时，查看最近变更并推断主要风险面。
   - 若用户要的是模块拆分、状态流、目录结构或迁移步骤，建议改用 **`/fec-plan`**。

2. 使用 `fec-testing-strategy` Skill 建立测试分层矩阵：
   - 静态检查
   - 单元测试
   - 组件测试
   - 轻量集成测试
   - E2E 测试
   - 视觉 / Storybook / interaction
   - a11y / 安全 / 性能等专项质量验证

3. 对每个风险给出建议测试层、优先级、建议命令、责任 skill 和失败定位证据。

4. 标明哪些风险只需要现有门禁覆盖，哪些需要新增测试，哪些暂不覆盖并说明原因。

5. 将计划保存到 `reports/test-plan-YYYY-MM-DD-HHmmss.md`：

   ```markdown
   # 前端测试计划

   > 生成时间: YYYY-MM-DD HH:mm
   > 范围: ...

   ## 风险概览

   | Risk | Impact | Suggested Layer | Priority | Evidence |
   | ---- | ------ | --------------- | -------- | -------- |

   ## 覆盖矩阵

   | Layer | What to Cover | Tooling / Command | Skill |
   | ----- | ------------- | ----------------- | ----- |

   ## 执行顺序

   ## 不覆盖项与原因
   ```

6. 若用户要求继续实现测试，按矩阵分流到对应专项 skill 或 agent。

## 边界

- 不设计组件架构、目录结构、状态流或迁移阶段；这些属于 `/fec-plan` 或 `fec-architect`。
- 不默认要求补齐完整测试金字塔，只按风险和维护成本选择覆盖。
- 不负责大规模编写测试文件；用户明确要求落地时再分流到专项测试 skill。
