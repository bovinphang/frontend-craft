# 图表模式

当用户要求某类具名技术图时，使用这些约定。

## 架构图

- 按层级分组节点：client、edge、service、data、external。
- 边界重要时，使用泳道或分组容器。
- 分层块状架构图优先使用水平层、左侧层标签、按内容收紧的容器，以及用于 security、monitoring 或 governance 的可选横切侧边栏。
- 将 bus 或 event-stream 节点放在 publisher/subscriber 行的中心附近。
- 数据库使用圆柱，服务使用圆角矩形，外部系统使用虚线外框。
- 只有当 data flow 或 dependency direction 重要时才使用箭头；纯能力栈优先用空间分组表达。

## ERD

- 将表建模为容器，列以行展示。
- 在标签中标记主键和外键。
- 基数关系不完全明确时，使用虚线关系边。
- 包含多个关系簇的 schema 优先使用自上而下布局。
- 使用稳定的表宽和行高，让长列名换行，而不是撑开相邻表格。

## UML 类图

- 使用拆分为名称、字段和方法的类框。
- 继承使用空心三角箭头，接口实现使用虚线空心三角箭头。
- 接口或抽象基类放在具体类上方。
- 关联关系保持正交布线，并将 multiplicity 标签放在其描述的边段附近。

## 序列图

- 参与者从左到右排列，时间从上到下推进。
- 请求、响应和异步消息的样式保持可视化区分。
- 只有能澄清嵌套工作时才添加 activation box。
- 为消息标签留出足够水平间距；如果标签碰撞，换行或将交互拆成多个阶段。

## 机器学习 / 深度学习

- tensor shape 重要时，在第二行展示。
- 对 encoder、decoder、embedding、attention 和 output 阶段分组。
- skip connection 绕开主流程布线。

## 流程图

- 开始/结束使用椭圆，过程使用矩形，决策使用菱形。
- 为决策分支添加标签。
- 尽可能将分支合并回中心路径。
- 主路径保持纵向。备选或错误分支横向绕出，再用正交边回到主路径。
- 可编辑 draw.io 流程图最终交付前遵循 [flowchart-quality.md](flowchart-quality.md)。

## 思维导图

- 将中心概念放在画布中央，一层分支分布到不同象限。
- 使用无箭头曲线边，并保持每条分支从父节点到叶子的一致配色。
- 叶子节点从中心向外展开；不要让分支穿过中心标签。

## 网络拓扑

- 使用 internet、edge、switching/routing 和 devices 等层级行。
- 有线和无线连接应有可视区分，通常实线表示有线，虚线表示无线。
- 终端设备按行或列对齐，让连接路径易于扫读。
