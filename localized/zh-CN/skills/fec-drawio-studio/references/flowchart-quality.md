# 流程图质量

当 draw.io / diagrams.net 交付物是流程、工作流、审批路径、发布路径、事故响应、入职流程或其他需要打开后依然可读的流程图时，使用本参考。

## 可读性约定

- 手工编写的流程图保持窄画布。优先使用 600 到 800 的 `pageWidth`；除非图表明确是横向跨职能表，否则不要超过 800。
- 将 `dx` 和 `dy` 设置为接近页面尺寸，避免 diagrams.net 初始缩放后文字变得很小。
- 主路径自上而下推进。备选、失败、重试和异常分支可以横向绕出，但需要用正交连接线回到主路径。
- 超过五个密集阶段时，优先拆成多张图或多个页面，而不是做成一个超长超宽画布。

## 阶段分组

- 真实阶段、角色、阶段组或边界上下文使用 `swimlane` 容器；当标题和包含关系重要时，不要用松散矩形代替。
- 阶段容器保持 10px 网格对齐，通常保留 20px 页面外边距，阶段之间保留 30-40px 垂直呼吸空间。
- 容器标题建立清晰层级：阶段标签约 14-16px、加粗，并使用明确边框色。
- 阶段容器设置 `connectable="0"` 和 `collapsible=0`，避免连接线误连容器本身，也避免出现折叠控件。
- 容器尺寸应包住子节点并保留至少 15px 底部留白。单侧大面积空白通常意味着需要重新布局。

## 标签

- 过程框文字默认居中；只有表格类内容确实需要行对齐时才改用其他对齐。
- 过程节点使用 `whiteSpace=wrap;html=1`，普通正文不小于 12px。
- 节点标题放在最前，并比正文更醒目。标签保持简洁，把过长实现细节放进备注或 metadata。
- 图表语言与用户请求一致，保存前转义 XML 属性内容。

## 连接线

- 常规流程图连接线使用 `edgeStyle=orthogonalEdgeStyle`。
- 阶段到阶段的主路径应比次要边更醒目，例如使用更粗线宽和一致强调色。
- 决策分支和异常路径需要标注，标签应靠近它描述的边段。
- 一个节点扇出到多个同级节点时，使用明确的 `entryX`、`entryY`、`exitX` 和 `exitY` 锚点，避免分支互相穿越。
- 重试、回滚和循环连接线通过左侧或右侧边距及显式 waypoint 绕行，不要斜穿内容。

## 视觉系统

- 使用小型色板，每种阶段类型或角色组一个颜色。过程节点填充保持中性，由边框或标题承载阶段色。
- 警告、失败、待定和未来状态只用于异常节点；不要把状态色当装饰。
- 结构性阶段边界使用方角容器；普通过程节点可以使用圆角。

## 最终检查

交付前运行：

```bash
node skills/fec-drawio-studio/scripts/diagram-lint.mjs diagram.drawio --format markdown --strict
```

然后手动打开图表，检查主路径连线没有交叉、标签没有截断、阶段没有像折叠态一样不可读，并且 diagrams.net 默认缩放下文字仍然清晰。
